# 复盘 — RAG 课程助手

## 认知变化

一开始我以为 RAG 的重点是“能不能把相似的词匹配出来”。做完这次测试后，我发现更关键的是“哪些匹配可以被当成依据”。如果检索阶段把弱相关词也算进去，比如“一个”“没有”“day1”，系统就会把资料外问题误判成资料内问题，然后给出看起来合理但其实没有依据的回答。

所以我的理解从“扩大召回”变成了“控制证据”。RAG 不是把所有沾边的内容都找出来，而是要让回答可以追溯到明确的 FAQ chunk。来源编号不是装饰，它是判断回答是否可复核的证据。

## 遇到的困难

第一个困难是中文关键词检索噪声很大。原来的 n-gram 切词能生成很多词片段，但很多片段没有真实语义，会让错误 FAQ 排到前面。例如混淆问题“说一个Day1没有教的内容”，如果只看到 `day1` 或 `没有`，系统就容易误召回课程 FAQ。

第二个困难是跨段落问题需要多个 FAQ 一起支撑。比如“Spec中的非目标和AI过度设计有什么关系？”同时涉及 `[faq-04]` 的 Spec 三要素和 `[faq-10]` 的非目标说明。只靠单个词命中不够，需要让“非目标”“过度设计”“数据库”“Web UI”这些核心词真正影响排序。

第三个困难是 CLI 路径和测试路径不一致。测试脚本会自己加载 chunks，但 `main.py` 原来传的是空列表，导致实际命令行运行时永远找不到资料。这个问题提醒我：测试通过不等于用户入口一定可用，必须单独做 CLI 冒烟测试。

## 我的判断与取舍

我没有选择修改测试用例来让结果变好，因为那样不能证明系统真的完成了目标。我的判断是：测试问题本身覆盖了资料内、资料外、跨段落和混淆场景，应该修实现，而不是降低验收标准。

我也没有引入向量数据库或分词库。项目的非目标明确要求不引入数据库，课程目标是理解 RAG 基本流程，所以我选择在现有关键词检索基础上做小范围改进：课程领域词表、意图扩展、标题加权和明确拒答规则。这样改动可解释，也符合项目边界。

另外，我保留了 mock LLM 的调用接口，但最终回答以检索到的 FAQ chunk 为准。原因是 mock 服务本身会重新粗略匹配 prompt，可能把已经检索好的结果打乱。对这个项目来说，“来源可复核”比“回答更像自然语言”更重要。

## 改进结果

本次改进后：

- `main.py` 会真实加载 `data/course-faq.md`。
- `retrieve.py` 不再依赖大量弱 n-gram，而是优先匹配课程领域关键词。
- `answer.py` 对资料外和混淆问题先拒答，避免无依据生成。
- `docs/test-record.md` 记录了每个测试的输入、预期和实际。
- `README.md` 提供了安装、运行和测试命令，方便同桌或助教复现。

## 后续改进方向

如果继续改进，我会优先做两件事：

1. 把检索命中的关键词和分数输出成调试信息，方便定位为什么某个 FAQ 被召回。
2. 把拒答规则从硬编码模式整理成更清晰的函数或配置，让未来新增测试问题时更容易维护。

这次最大的收获是：RAG 的质量不只看能不能回答，还要看能不能证明“为什么这样回答”。可复核比看起来聪明更重要。
