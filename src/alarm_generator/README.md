# alarm_generator

课题 A：告警数据生成模块。

职责：

- 基于 `data/samples/alarms.json` 的真实格式，通过场景模板生成扩展告警数据。
- 输出到 `data/generated/`。
- 生成的数据字段应尽量保持与原始 `alarms.json` 一致。

注意：

- 第一优先级是配合 `perception_agent` 解析真实样例。
- 数据生成是后续扩展，不要改变原始样例文件。

---

## 基于真实样本的数据扩展方案

当前 `alarms.json` 中真实格式告警数量有限，需扩展数据以支持全量规则的充分测试。

### 扩展策略（场景驱动，非孤立生成）

```text
1. 定义典型故障场景模板（7 个）
   - 线路跳闸（开关+刀闸联动）
   - 线路跳闸后重合成功
   - 保护通道A网中断（双套同时）
   - 测控A网中断后复归
   - 保护动作后复归
   - 多开关联动分闸
   - 线路跳闸+保护通道异常
        ↓
2. 随机选择场景模板 + 厂站 + 设备编号
   每个场景内部按时间间隔生成 2~4 条连续告警
        ↓
3. 场景完整性优先：放得下完整场景才加入，不截断
   alarm_id 确认加入后分配，保持连续
        ↓
4. 时间锚定在 2023-08-29（与原始数据对齐）
   全局按时间排序后输出
```

### 模板参数池

| 参数类型 | 候选值 |
|---|---|
| 厂站 | `20kV示范变电站`、`20kV新华变电站`、`20kV城东变电站`、`35kV城南变电站`、`10kV工业园变电站` |
| 设备编号前缀 | `101`~`103`、`201`~`203`、`301`~`302`、`401`~`402` |
| 设备类型（不加前缀） | `开关`、`刀闸`、`接地刀闸` |
| 设备类型（加前缀） | `线路1保护`、`线路2保护`、`线路1测控`、`线路2测控` |

命名规范：开关/刀闸不加 kV 前缀（`101开关`），线路保护/测控加 kV 前缀（`10kV101线路1保护`）。

### 生成示例（场景内连续爆发）

```text
场景: 线路跳闸后重合成功 (interval=1s)
  2023年08月29日 04:51:04 20kV城东变电站 20kV201线路1保护 动作(全数据判定)
  2023年08月29日 04:51:05 20kV城东变电站 201开关 分闸
  2023年08月29日 04:51:06 20kV城东变电站 201开关 合闸
```

### 运行方式

```powershell
# 生成 50 条扩展数据（默认，从 data/samples/alarms.json 读取）
python src/alarm_generator/alarm_generator.py

# 自定义数量和种子
python src/alarm_generator/alarm_generator.py --count 200 --seed 42
```

### 输出格式

扩展数据与原始数据合并输出到 `data/generated/extended_alarms.json`：

```json
{
  "description": "告警采样数据（含模拟扩展）",
  "total_count": "<原始数 + 生成数>",
  "original_count": "<原始条数>",
  "generated_count": "<生成条数>",
  "alarms": [
    // ...原始 N 条 + 生成的 M 条
  ]
}
```

生成告警的 `alarm_id` 为 `ALM-GEN-0001` ~ `ALM-GEN-NNNN`。

### 注意事项

- 生成数据**不覆盖**原始 `alarms.json`。
- 输出到 `data/generated/extended_alarms.json`。
- 可先用扩展数据跑 `perception_agent` 验证解析鲁棒性，再交给下游 B 模块。
- 后续可根据 `features.json` 中 `signal_mappings` 的 `object_name` 定向构造匹配告警。
