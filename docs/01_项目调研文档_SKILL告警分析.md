# 项目调研文档：基于 AI Agent 的电网告警事件分析系统

整理日期：2026-07-07  
当前方向：告警事件分析、SKILL 规则管理、多智能体编排、表达式引擎

## 1. 项目背景

电力系统运行过程中会产生大量告警信息。单条告警往往不能直接说明故障类型，真实业务中通常需要在一个时间窗口内综合多条告警，结合设备、时间、等级、告警类型和规则组合，判断是否形成某类事件。

传统告警事件识别主要依赖人工编写规则或固定模板，存在以下问题：

- 规则维护成本高。
- 新场景扩展困难。
- 大量告警涌入时，关键事件容易被淹没。
- 粗规则容易误报，细规则容易漏报。
- 推送过多低价值告警会增加调度人员负担。

本项目希望引入 AI Agent、SKILL 规则管理、表达式引擎和多智能体编排能力，构建一个可演示的电网告警事件分析系统。

## 2. 当前项目定位

项目名称建议：

```text
基于 AI Agent 的电网告警事件分析与智能匹配系统
```

核心目标：

```text
模拟告警数据
  -> 感知预处理
  -> SKILL 规则加载
  -> 表达式解析与求值
  -> Agent 语义理解与判定
  -> 标准事件生成
  -> 前端可视化展示
```

当前项目不再以图像、视频、多模态违规操作识别作为主线。核心输入是告警数据、规则定义、特征表达式和模拟数据。

## 3. 核心业务理解

### 3.1 告警

告警是电力系统中由监控、采集、保护或运行系统产生的异常提示，例如电流越限、保护动作、开关跳闸、通信中断等。

### 3.2 事件

事件不是单条告警，而是一段时间内多条告警按照一定规则组合后形成的业务判断。

示例：

```text
同一线路在 30 秒内出现：
线路电流越限 + 保护动作 + 开关跳闸

则可能形成：
线路故障事件
```

### 3.3 规则

规则用于描述告警组合与事件之间的关系，形式可能类似：

```text
A & B & C within 30s on same_device => 事件 D
```

规则可能包含：

- 告警类型约束。
- 时间窗口约束。
- 同设备约束。
- 告警顺序约束。
- 次数统计约束。
- 事件等级。
- 推送优先级。

## 4. 参考方向调研

### 4.1 复杂事件处理 CEP

复杂事件处理用于从连续事件流中识别有意义的事件模式。其思想与本项目高度相关。

可借鉴点：

- 将告警看作事件流。
- 用时间窗口聚合事件。
- 用条件表达式描述事件模式。
- 支持 AND、OR、NOT、before、within、count 等逻辑。
- 输出标准化事件。

本项目可借鉴 CEP 的事件流处理思想，但不一定直接引入复杂 CEP 框架，第一版可以自己实现轻量表达式引擎。

### 4.2 规则引擎

传统规则引擎通常通过结构化规则完成判断，例如 Drools、Easy Rules 等。

可借鉴点：

- 规则与业务代码分离。
- 规则可配置、可扩展。
- 规则执行结果可解释。
- 支持规则优先级。

本项目与传统规则引擎的区别在于：规则不只是程序配置，还会被封装成 SKILL 文件，由 Agent 按需加载并结合大模型理解规则语义。

### 4.3 Agent 工作流

Agent 工作流适合处理多步骤任务，例如解析输入、加载工具、调用规则、生成报告。

本项目可设计多个智能体：

- 感知智能体：负责告警数据清洗、结构化和初步分类。
- 规则 SKILL 智能体：负责查找和加载候选 SKILL。
- 分析智能体：负责规则表达式解析、求值和语义判断。
- 生成智能体：负责生成标准事件和分析报告。
- 调度智能体：负责编排整体流程。

### 4.4 SKILL 机制

本项目采用 SKILL 方式管理告警识别模式和事件规则。每条特征定义或事件规则都封装为独立 SKILL 文件，Agent 根据输入上下文按需加载。

相对传统 RAG 的优势：

- 规则边界更清晰。
- 可解释性更强。
- 减少向量检索误召回。
- 每条规则可以独立维护。
- Agent 可直接读取规则语义、表达式和输出要求。

### 4.5 时序数据仿真

由于企业真实数据有限且需要脱敏，本项目需要构建模拟告警数据生成引擎。

可借鉴点：

- 基于场景模板生成告警。
- 加入正常噪声和干扰告警。
- 支持多事件并发。
- 生成标准答案 labels。
- 用于测试规则识别准确率。

## 5. 当前技术路线

```text
课题 A：模拟告警数据生成与感知预处理
  输出结构化告警序列和初步特征

课题 B：SKILL 规则管理与智能匹配
  输出候选 SKILL 和规则语义

课题 C：规则分析与表达式引擎
  输出规则求值结果和事件触发判断

课题 D：事件生成、调度编排与可视化
  输出标准事件、报告和前端展示
```

## 6. 与传统方案对比

| 方案 | 特点 | 问题 | 本项目改进 |
|---|---|---|---|
| 人工硬编码规则 | 稳定、可控 | 扩展困难、维护成本高 | 使用 SKILL 文件管理规则 |
| 普通关键词匹配 | 简单 | 不能理解时序和组合关系 | 使用表达式引擎处理时序规则 |
| 传统 RAG 检索 | 适合知识问答 | 规则匹配可能误召回 | 用 SKILL 精确加载规则 |
| 单 Agent 判断 | 实现快 | 逻辑混杂、难解释 | 多智能体分工编排 |
| 纯大模型判断 | 语义理解强 | 不稳定、难复现 | 程序求值 + 大模型解释结合 |

## 7. 项目创新点

1. **SKILL 化规则管理**  
   将告警特征和事件规则封装为可加载 SKILL 文件，替代传统 RAG 向量检索主流程。

2. **时序告警表达式引擎**  
   支持时间窗口、同设备约束、告警组合、告警顺序和计数条件。

3. **多智能体协同分析**  
   通过感知、规则、分析、生成、调度等智能体完成端到端事件识别。

4. **模拟数据生成引擎**  
   基于真实电网场景构造告警序列，解决真实数据不足和脱敏限制问题。

5. **可解释事件输出**  
   输出命中规则、匹配条件、告警证据、事件等级和推送建议。

## 8. 风险与应对

| 风险 | 说明 | 应对 |
|---|---|---|
| 真实数据不足 | 企业样本可能有限 | 设计模拟数据生成器 |
| 规则格式不确定 | 企业规则可能较复杂 | 先支持基础表达式，再逐步扩展 |
| 大模型判断不稳定 | 纯模型推理可能幻觉 | 确定性规则由表达式引擎执行 |
| SKILL 设计不统一 | 规则文件可能混乱 | 制定统一 SKILL 模板 |
| 前后端工作量大 | 时间有限 | 优先完成演示闭环 |

## 9. 调研结论

本项目的最佳实施路线不是多模态视频识别，也不是普通 RAG 问答，而是：

```text
模拟时序告警数据 + SKILL 规则管理 + 表达式引擎 + 多智能体编排 + 标准事件生成
```

该路线既符合企业导师提出的真实需求，也适合项目式实习答辩和 Agent 开发岗简历表达。
