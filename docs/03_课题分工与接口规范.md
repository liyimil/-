# 课题分工与接口规范

整理日期：2026-07-13  
当前依据：`data/samples/samples-md` 数据集

## 1. 分工总览

| 课题 | 名称 | 核心职责 | 输出 |
|---|---|---|---|
| A | 告警数据感知与预处理 | 解析 `alarms.json` 和 `signal_name` | 结构化告警 |
| B | SKILL 规则管理与智能匹配 | 管理 Feature SKILL / Rule SKILL | 候选特征和候选规则 |
| C | 规则分析与表达式引擎 | 解析并求值 `expression` | 特征/规则判定结果 |
| D | 事件生成、调度编排与可视化 | 编排流程并生成事件 | 标准事件和展示结果 |

## 2. 每个人先做什么

| 课题 | 第一阶段任务 | 输入 | 输出 |
|---|---|---|---|
| A | 读取 `alarms.json`，解析 `signal_name` 中的时间、厂站、设备、动作 | `data/samples/samples-md/alarms.json` | 结构化告警 JSON |
| B | 读取 `features.json` / `rules.json`，封装 Feature SKILL 和 Rule SKILL，给出 `feature_states` | A 的结构化告警、`features.json`、`rules.json` | 候选 SKILL、特征状态 |
| C | 实现表达式解析与求值，先跑通 `(@1|@2)&@3&(@4|@5|@6)` | B 的 `feature_states`、Rule SKILL | 规则是否触发 |
| D | 串联 A/B/C，生成标准事件并做简单展示 | A/B/C 输出 | 标准事件 JSON / 页面展示 |

## 3. 总流程

```text
A 读取 alarms.json 并解析 signal_name
  -> B 根据 features.json / rules.json 加载候选 SKILL
  -> C 解析 feature.expression / rule.expression
  -> C 输出规则触发结果
  -> D 根据 event_type / event_level / output_format 生成标准事件
```

## 8. SKILL 文件规范

### 8.1 Feature SKILL

```markdown
# FEATURE SKILL: 特征名称

## metadata
- skill_id:
- source_feature_id:
- type: feature
- scope:
- valid_time:

## expression
@1=1

## signal_mappings
列出 object_type、object_name、signal_feature、feature_desc。

## explanation
解释该特征代表什么业务含义。
```

### 8.2 Rule SKILL

```markdown
# RULE SKILL: 规则名称

## metadata
- skill_id:
- source_rule_id:
- type: event_rule
- event_level:
- event_type:
- valid_time:
- enabled:

## expression
规则表达式。

## output_format
事件输出模板。

## explanation
解释规则触发后的业务含义。
```

## 9. 表达式语法约定

第一版支持：

```text
@1=1
@1&@2
@1|@2
!@1
(@1|@2)&@3
(@1|@2)&@3&(@4|@5|@6)
```

后续根据企业确认结果，再扩展：

```text
valid_time 时间窗口
scope 作用域
output_format 变量替换
```

## 10. 已确认映射关系

```text
rules.expression 中的 @n
  -> features.json 中 feature_id 为 @n 的特征
```

例如：

```text
rule_id=5
expression=(@1|@2)&@3&(@4|@5|@6)
```

表示：

```text
特征 @1 或 @2 成立
且特征 @3 成立
且特征 @4/@5/@6 中任一成立
```

因此 B/C 模块之间的关键接口应包含：

```json
{
  "feature_states": {
    "@1": true,
    "@2": false,
    "@3": true,
    "@4": true,
    "@5": false,
    "@6": false
  }
}
```

后续仍需确认 `features.expression` 与 `signal_mappings` 的具体触发方式。

## 11. 第一阶段统一案例

先围绕 `rule_id=5` 跑通链路：

```text
rule_name: [精准]20kV线路故障（远方手合）
expression: (@1|@2)&@3&(@4|@5|@6)
```

老师已确认：规则里的 `@1~@6` 对应 `features.json` 中 `feature_id=@1~@6` 的六个特征。
