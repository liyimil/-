# 课题分工与接口规范

整理日期：2026-07-13  
当前依据：`data/samples/samples-md` 数据集

## 1. 分工总览

| 课题 | 名称 | 核心职责 | 输出 |
|---|---|---|---|
| A | 告警数据感知与预处理 | 解析 `alarms.json` 和 `signal_name` | 结构化告警 |
| B | SKILL 规则管理与智能匹配 | 管理 Feature SKILL / Rule SKILL | 候选特征和候选规则 |
| C | 规则分析与表达式引擎 | 解析并求值 `expression` | 特征/规则判定结果 |
| D | 事件生成、调度编排与可视化 | 编排流程并生成事件 | 标准事件和展示结果 |

## 2. 每个人负责什么

| 课题 | 任务 | 主要修改目录 | 输入 | 输出 |
|---|---|---|---|---|
| A | 读取 `alarms.json`，解析 `signal_name` 中的时间、厂站、设备、动作 | `src/perception_agent/`、`src/alarm_generator/`、`data/generated/` | `data/samples/samples-md/alarms.json` | 结构化告警 JSON |
| B | 读取 `features.json` / `rules.json`，封装 SKILL；用告警匹配 `signal_mappings` | `src/skill_engine/`、`skills/` | A 的结构化告警、`features.json`、`rules.json` | 候选 SKILL、局部信号状态 |
| C | 求值 `feature.expression` 和全量 `rule.expression` | `src/expression_engine/`、`tests/rule_cases/` | B 的 `signal_mapping_states`、Feature SKILL、Rule SKILL | 特征状态、规则是否触发 |
| D | 串联 A/B/C，生成标准事件并做简单展示 | `src/orchestrator/`、`src/event_generator/`、`frontend/` | A/B/C 输出 | 标准事件 JSON / 页面展示 |

## 3. 模块交付链路

各模块按顺序交付 JSON 结果，后一模块只依赖前一模块的标准输出，不直接依赖对方内部实现。

| 交付方 | 接收方 | 交付内容 | 建议落盘位置 | 说明 |
|---|---|---|---|---|
| A | B | `structured_alarms` | `data/generated/structured_alarms.json` | A 负责把原始 `alarms.json` 清洗、解析为结构化告警，B 直接读取该结果做语义匹配。 |
| B | C | `skill_match` | `data/generated/skill_match.json` | B 负责加载 Feature/Rule SKILL，并输出候选 SKILL 与 `signal_mapping_states`，C 基于这些状态做表达式求值。 |
| C | D | `expression_result` | `data/generated/expression_result.json` | C 负责输出 `feature_states`、`feature_hits`、`rule_results`，D 只根据规则判定结果生成标准事件。 |
| D | 前端/报告 | `orchestration_result` / `events` | `data/generated/orchestration_result.json` | D 负责基于 QwenPaw 多智能体流程做整体编排，调用事件生成模块，并把结果整理成前端和答辩可展示的数据。 |

当前 D 模块可先使用 `tests/mock_outputs/d_module_mock.json` 模拟 A/B/C 输出跑通链路。后续 A/B/C 真实结果完成后，只替换输入 JSON 或 adapter 数据来源，事件生成逻辑保持稳定。

## 4. 总流程

```text
A 读取 alarms.json 并解析 signal_name
  -> B 用 signal_mappings 做运行时语义匹配
  -> C 解析 feature.expression 得到 feature_states
  -> C 解析 rule.expression 得到 rule_results
  -> C 输出规则触发结果
  -> D 根据 event_type / event_level / output_format 生成标准事件
```

## 9. SKILL 文件规范

### 9.1 Feature SKILL

```markdown
# FEATURE SKILL: 特征名称

## metadata
- skill_id:
- source_feature_id:
- type: feature
- scope:
- valid_time:

## expression
@1=1

## signal_mappings
列出 object_type、object_name、signal_feature、feature_desc。

## explanation
解释该特征代表什么业务含义。
```

### 9.2 Rule SKILL

```markdown
# RULE SKILL: 规则名称

## metadata
- skill_id:
- source_rule_id:
- type: event_rule
- event_level:
- event_type:
- valid_time:
- enabled:

## expression
规则表达式。

## output_format
事件输出模板。

## explanation
解释规则触发后的业务含义。
```

## 10. 表达式语法约定

第一版支持：

```text
@1=1
@1&@2
@1|@2
!@1
(@1|@2)&@3
(@1|@2)&@3&(@4|@5|@6)
```

后续根据企业确认结果，再扩展：

```text
valid_time 时间窗口
scope 作用域
output_format 变量替换
```

## 11. 已确认映射关系

```text
rules.expression 中的 @n
  -> features.json 中 feature_id 为 @n 的特征
```

例如：

```text
rule_id=5
expression=(@1|@2)&@3&(@4|@5|@6)
```

表示：

```text
特征 @1 或 @2 成立
且特征 @3 成立
且特征 @4/@5/@6 中任一成立
```

因此 B/C 模块之间的关键接口应包含：

```json
{
  "feature_states": {
    "@1": true,
    "@2": false,
    "@3": true,
    "@4": true,
    "@5": false,
    "@6": false
  }
}
```

已确认：`feature.expression` 中的 `@n` 对应该 feature 内部 `signal_mappings.index=@n`。后续只需细化 `alarm.signal_name` 与 signal_mapping 各字段的语义匹配策略。

同时要区分两层 `@n`：

| 层级 | `@n` 含义 |
|---|---|
| `rule.expression` | 全局特征编号，对应 `features.feature_id=@n` |
| `feature.expression` | feature 内部局部 signal_mapping 索引，对应该 feature 的 `signal_mappings.index=@n` |

## 12. 规则处理目标

系统应支持 `rules.json` 中 461 条规则的统一加载、索引和表达式求值。

Rule 5 仅作为示例：

```text
rule_name: [精准]20kV线路故障（远方手合）
expression: (@1|@2)&@3&(@4|@5|@6)
```

老师已确认：规则里的 `@1~@6` 对应 `features.json` 中 `feature_id=@1~@6` 的六个特征。
